# 纳川教学视频 · 准高中生AI学习伙伴

## 定位说明

**纳川教学视频**是为准高中生和家长打造的AI学习伙伴，致力于"让抽象概念动起来"。

- **目标用户**：准高中生（初三升高一）+ 家长
- **学科覆盖**：初中全科 + 高中预科（物理、数学、化学、生物、语文、英语）
- **教学理念**：让抽象概念动起来 — 用动态可视化把难懂的知识点讲透
- **视频时长**：默认 60-90 秒，适合短视频学习场景，碎片时间高效吸收
- **语言风格**：亲切、清晰、鼓励性 — 像一位耐心的学长/学姐在讲解
- **开箱即用**：零 API 配置即可生成完整视频（系统内置配音 + 本地 HyperFrames 渲染，macOS / Windows / Linux 全平台兼容）

---

## 概述

输入一个概念（例："声现象"）→ 按 8 阶段管线生产 → 两种动态产出，**都是 HyperFrames 本地渲染的 MP4，共用同一个 index.html**：

| 产出 | 规格 | 用途 |
|---|---|---|
| **配音视频** | 1080p、60-90s、中文旁白 + 字幕 | 完整讲解，发视频号 / 给孩子看 |
| **无声动图** | 1080p、~32s、无声、循环 | 公众号内嵌、知识点快速回顾 |

关键：两种产出**共用同一个 `index.html`**，靠 HyperFrames 变量 `mode`（video/silent）切换——
搭一次内容，两种都能出。silent 模式紧凑时间轴、无字幕、不淡出（循环无缝），质感和动效
与视频版完全一致。

共用：**7 段结构**（1 标题 + 5 子概念 + 1 总结）、**同一套主题配色**
（`references/color-palettes.md`）、**同一份分镜脚本**（`references/storyboard-guide.md`）。

## 阶段 0：研究与拆解（Research）

**目标**：在动笔之前，把知识点彻底搞懂，确保内容准确、有深度、有针对性。

### 输入
- 用户给定的学科概念（如"声现象""杠杆原理"）
- 适用学段（默认：初二物理）

### 执行步骤

**0.1 知识点核对**
- 翻查对应学段的教材目录，确认这个概念的标准表述和知识边界
- 确认前置知识和后续知识（这个概念是从哪来的、往哪去）
- 标出核心公式、常量、单位（声速 340m/s、光速 3×10⁸m/s 等）

**0.2 常见误解挖掘**
- 搜索这个知识点学生最容易搞错的地方（"[概念] 常见错误""[概念] 误区"）
- 找出 2-3 个真实的常见误解（不是编造的）
- 这些误解就是后续"误解优先法"的弹药

**0.3 生活场景联想**
- 找出 2-3 个学生日常生活中能接触到的相关现象
- 这些是钩子和应用段的素材来源

**0.4 确定概念拆解**
- 把核心概念拆成 5 个子概念（对应 7 段结构中的 s2-s6）
- 子概念按教材逻辑排序：是什么 → 怎么工作 → 有什么特性 → 怎么用
- 每个子概念配一个核心视觉意象

### 输出
- 知识点清单（核心定义 + 公式常量 + 边界）
- 常见误解清单（2-3 个，附正确解释）
- 生活场景清单（2-3 个）
- 5 个子概念拆解草案

### 质量门 G0：研究深度检查

> **必须全部通过才能进入阶段 1。**

| 检查项 | 通过标准 | 未通过处理 |
|---|---|---|
| 知识点准确性 | 核心定义与教材一致，公式常量正确 | 重新查证，直到准确 |
| 误解数量 | 至少找到 1 个真实常见误解 | 继续搜索；确实没有则标记"无典型误解" |
| 子概念拆解 | 5 个子概念，逻辑递进，不重叠 | 调整拆解方案 |
| 生活场景 | 至少 1 个能引起共鸣的日常场景 | 继续联想，从学生视角出发 |

> 详见 `references/teaching-narrative.md` 的"误解优先法"和"引导式发现法"章节。

## 阶段 1：概念提案（Proposal）

**目标**：确定视频的核心钩子、叙事策略和整体结构，在动笔前对齐方向。

### 输入
- 阶段 0 的研究成果（知识点 + 误解 + 场景 + 子概念拆解）

### 执行步骤

**1.1 选定钩子策略（默认：误解优先）**

如果阶段 0 找到了真实常见误解 → **默认用误解优先法开头**：
- 钩子句式："很多人以为……但其实不是这样。"
- 这是经过验证的最高效教学视频开头模式

如果没有典型误解 → 从以下钩子类型中选最匹配的：

| 钩子类型 | 模式 | 适用场景 |
|---|---|---|
| **反直觉型** | "你以为是这样……其实刚好相反。" | 物理、生物有反直觉结论的主题 |
| **问题型** | "为什么会……？" | 所有学科，最通用 |
| **数字冲击型** | "你知道吗？X 的速度是每秒 3 亿米。" | 有具体数字的主题 |
| **生活场景型** | "你有没有注意到……？" | 贴近生活的主题 |

> 钩子写作原则详见 `references/teaching-narrative.md` 第六节"钩子类型选择"。

**1.2 确定叙事弧线**

根据内容选择 7 段弧线的具体角色分配：

| 弧线类型 | 适用场景 | s1-s7 角色分配 |
|---|---|---|
| **误解优先型**（默认） | 有典型误解的知识点 | s1=抛误解 → s2=展示误解 → s3=指出错误 → s4=正确概念 → s5=核心洞察 → s6=应用巩固 → s7=总结回扣 |
| **引导发现型** | 需要推理过程的概念 | s1=提问题 → s2=朴素尝试 → s3=遇到困难 → s4=关键洞察 → s5=逐步构建 → s6=推广应用 → s7=一般化总结 |
| **递进讲解型** | 层层深入的系统性知识 | s1=总览 → s2=基础概念 → s3=进阶1 → s4=进阶2 → s5=核心机制 → s6=综合应用 → s7=总结 |

**1.3 选定配色方案**

根据主题从 `references/color-palettes.md` 的 7 套调色板中选择：
- 声学/波动 → sound（cream + 橙）
- 光学/颜色 → light（暖黄 + 蓝）
- 力学/运动 → mechanics（暖红 + 灰）
- 电学/电路 → electricity（靛蓝 + 金）
- 热学/能量 → heat（砖红 + 米）
- 生物/生命 → biology（墨绿 + 米）
- 数学/逻辑 → math（深蓝 + 金）

**1.4 选择配音引擎**

- 默认 auto：自动按五级 fallback 链选最佳引擎（Minimax → macOS say → Windows SAPI → pyttsx3 → 纯文本估算）
- 全平台零配置可用，Windows 用户自动走 PowerShell SAPI，Linux 自动走纯文本估算
- 如设置了 `MINIMAX_API_KEY` 环境变量 → 优先用 Minimax 高清 TTS（female-chengshu）

### 输出
- 钩子文案（1 句话，≤25 字）
- 弧线类型 + 7 段角色分配
- 选定的配色方案
- 配音引擎选择

### 质量门 G1：概念提案检查

> **必须全部通过才能进入阶段 2。**

| 检查项 | 通过标准 | 未通过处理 |
|---|---|---|
| 钩子有效性 | 前 5 秒能制造信息缺口或认知冲突 | 重写钩子，试试其他钩子类型 |
| 弧线合理性 | 7 段角色分配与内容类型匹配 | 调整弧线类型 |
| 钩子-结尾呼应 | 钩子提出的问题/冲突在 s7 能得到回答 | 调整弧线，确保首尾呼应 |
| 配色匹配度 | 配色与主题的自然联想一致 | 换一套调色板 |

> 详见 `references/teaching-narrative.md` 的"讲解型弧线模板"和"钩子类型选择"章节。

## 阶段 2：脚本撰写（Script）

**目标**：写出 7 段旁白文案，每段配画面描述，遵循 Mayer 多媒体学习原则。

### 输入
- 阶段 1 的概念提案（钩子 + 弧线 + 配色）
- 阶段 0 的知识点拆解

### 执行步骤

**2.1 写 7 段旁白**

按阶段 1 确定的弧线角色分配，为每一段写旁白文案。详细规范见 `references/storyboard-guide.md`。

核心规则（**必须遵守**）：
- 每段 30-50 字（中文口语约 4-5 字/秒，50 字 ≈ 11 秒）
- 口语化，像老师在讲，不是念课本
- **不读公式**：画面有公式，旁白用口语解释
- 数字读法写进文案（"3×10⁸" → "三亿米每秒"）
- 段落之间用"但是/所以"连接，不是"然后然后然后"
- 每段结尾自然收口，不要"接下来我们看"式的悬空

**2.2 写画面描述（visual 字段）**

每段旁白配一段画面描述，具体到：
- 左边/右边/中间分别是什么
- 什么元素先出现、什么后出现
- 关键动效是什么（绘制、滑入、弹出……）

这是阶段 3 分镜设计和阶段 5 场景编排的直接依据。

**2.3 生成 storyboard.json**

按 `references/storyboard-guide.md` 的格式组装成 JSON 文件。
每个 segment 必须包含：`id`, `title`, `narration`, `visual`, `transition`, `min_duration`

顶层字段：`topic`, `topic_slug`, `chapter`, `palette`, `voice`, `segments`

### 输出
- `storyboard.json`（7 段完整分镜脚本）

### 质量门 G2：脚本质量门（5 项检查）

> **必须全部通过才能进入阶段 3。这是第一道硬质量门。**

| # | 检查项 | 通过标准 | 未通过处理 |
|---|---|---|---|
| G2-1 | **知识点准确性** | 核心定义准确，公式/常量/单位正确，术语规范 | 查证后修正 |
| G2-2 | **结构完整性** | 7 段完整，子概念逻辑递进，首段有钩子，末段有收束 | 调整结构或补充内容 |
| G2-3 | **旁白质量** | 每段 30-50 字，口语化，不读公式，短句 | 精简或扩充文案 |
| G2-4 | **钩子有效性** | 前 5 秒制造信息缺口，不用"今天我们学习……" | 重写钩子 |
| G2-5 | **概念密度** | 全片新概念 ≤5 个，每个新概念有 ≥10s 讲解时间 | 砍掉超出的内容，留到下一期 |

> 完整检查清单详见 `references/quality-checklist.md` 第一节"脚本质量门"。
> 教学原则依据详见 `references/teaching-narrative.md` 的"Mayer 多媒体学习原则"。

## 阶段 3：分镜设计（Scene Plan）

**目标**：把脚本变成可执行的视觉蓝图——每个场景用五维分镜描述，确保动画有章法。

### 输入
- 阶段 2 的 `storyboard.json`
- 阶段 1 选定的配色方案

### 执行步骤

**3.1 五维分镜描述（每个场景必填）**

对 s1-s7 每个场景，逐一回答以下 5 个维度的问题。
**这不是可选的参考框架——每个场景都必须明确这五维。**

| 维度 | 含义 | 你要回答的问题 |
|---|---|---|
| **Subject 主题** | 画面的核心主体是什么 | 这个场景的视觉主角是什么？（示意图的哪个部分？卡片？数字？） |
| **Motion 动效** | 主体怎么动 | 入场方式？运动轨迹？关键帧时序？是绘制、滑入、弹出还是缩放？ |
| **Scene 场景** | 环境、布局、叠加层 | 整体布局是左图右卡还是居中？有哪些 overlay 元素（chip/callout/ghost）？ |
| **Spatial 空间构图** | 元素在画面中的位置关系 | 视觉焦点在哪？元素怎么分层（FG/MG/BG）？留白够吗？ |
| **Camera 镜头** | 视角和取景 | 是全景展示还是局部特写？有 zoom/pan 吗？（教学视频多为固定镜头，也要明确写出） |

> **animatic 思维**：在写分镜时，想象你在黑板上讲这道题——你会先写什么、再画什么、最后圈出什么？动画顺序要跟这个自然讲解顺序一致。

**3.2 转场规划**

按"有限转场家族原则"规划全片转场：
- **模糊叠化（blur-crossfade）**：默认转场，占 80% 以上
- **上推（push-up）**：仅用于有"章节推进"语义的地方，全片 ≤2 次
- 每段的 transition 字段写入 storyboard.json

**3.3 资产清单**

列出每个场景需要的视觉资产：
- SVG 示意图零件（从 `references/svg-parts.md` 选，或自行设计）
- 需要从模板/示例中复用的元素
- 特殊图标或图形

### 输出
- 每个场景的五维分镜描述（可以写在 storyboard.json 的 visual 字段扩充，或单独的分镜笔记）
- 全片转场方案
- 资产清单

### 质量门 G3：五维分镜检查

> **必须全部通过才能进入阶段 4。这是第二道硬质量门。**

| # | 检查项 | 通过标准 | 未通过处理 |
|---|---|---|---|
| G3-1 | **五维完整性** | 每个场景的 5 个维度都有明确描述，没有遗漏 | 补全缺失维度 |
| G3-2 | **主题一致性** | 画面内容与旁白讲的是同一个东西（时空同步） | 调整画面描述，使之与旁白对齐 |
| G3-3 | **动效有效性** | 每个动效都服务于知识点讲解，不是装饰 | 删除无效动效，替换为辅助理解的动效 |
| G3-4 | **信息密度** | 每场景 ≤3 个信息块（示意图 + 卡片 + 标注），不挤 | 拆分信息到多个场景，或简化元素 |
| G3-5 | **转场合理性** | 全片 ≤3 种转场，模糊叠化为主，push-up 有语义 | 统一转场方案 |

> 完整检查清单详见 `references/quality-checklist.md` 第二节"分镜质量门"。
> 动效设计原则详见 `references/motion-design.md`。
> 场景编写硬规则详见 `references/video-authoring.md`。

## 阶段 4：资产准备（Assets）

**目标**：生成所有需要的素材文件——配音、时间轴、配色变量。

### 输入
- 阶段 2 的 `storyboard.json`

### 执行步骤

**4.1 TTS 配音生成**

```bash
python3 scripts/tts.py storyboard.json --outdir audio/
```

- 自动检测：五级 fallback 链（Minimax → macOS say → Windows SAPI → pyttsx3 → 纯文本估算），auto 模式自动选最佳可用引擎
- 输出：`audio/seg-01.mp3` ~ `audio/seg-07.mp3` + `audio/durations.json`
- `durations.json` 是视频时间轴的**唯一来源**，后续所有时长数字都从这里来

**4.2 生成项目骨架**

```bash
python3 scripts/scaffold_video.py <project_dir>
```

## 阶段 5：场景编排（Edit / Scene Build）

**目标**：填充 HTML 场景内容，编写 GSAP 动效，让每个场景"活"起来。

### 输入
- 阶段 4 的 `index.html` 骨架
- 阶段 3 的五维分镜设计
- 阶段 4 准备的 SVG 素材

### 执行步骤

**5.1 填场景内容（s2-s6 子概念场景）**

每个子概念场景必须包含（缺一即空壳）：
1. **section-chip** — 左上角编号 + 小节名
2. **SVG 示意图** — `diagram-zone` 里的 SVG，≥3 个图形元素
3. **info-card** — 右侧要点卡片，2-4 行 + 可选公式行
4. **callout 标注** — 至少一条"指哪讲哪"
5. **ghost 编号** — 底纹编号，与 chip 同号

标题段（s1）：主题图标 SVG + 大标题 + 子概念胶囊排 + 副标题。
总结段（s7）：大数字/公式 + 小结卡片 + summary-bar 徽章排。

> 画面 ≠ 字幕：卡片写要点短语，字幕显示旁白原文，两者不重复。

**5.2 写入场编排（GSAP）**

必须遵循**动效四原则**（详见 `references/motion-design.md`）：

| 原则 | 实现要点 |
|---|---|
| **缓入缓出** | 入场用 ease out，位置移动用 ease in out；只有 opacity 和颜色可用 linear |
| **预备动作** | 关键元素入场前微微反向动一下再弹出（back.out ease） |
| **超出回弹** | 标题/数字用 back.out(1.5)；卡片用 power3.out（不回弹更稳重） |
| **分层错峰** | 多元素 stagger 0.09-0.25s，按逻辑顺序依次入场 |

每场景**至少 3 种 ease、至少 2 个入场方向**——全用 `y:30, opacity:0` = 没编排。

**5.3 加环境动效（停留帧 + 微动效）**

动画结束后不要立刻转场，让画面"停一下"：
- 关键信息停留 1.5-2.5s（给理解和记忆时间）
- 停留期间加一种细微环境动效（呼吸/漂移/波形/扩散）
- 一场景一种环境动效就够了，不要每个元素都在动

**5.4 音画同步检查**

- 讲到哪个元素，哪个元素入场（时空同步原则）
- 节奏：示意图先画（0.2-1.2s）→ callout 弹出 → 卡片滑入（旁白讲到要点时）
- 绝对禁止：画面讲 A 而旁白讲 B

### 输出
- 完整的 `index.html`（7 个场景全部填充完毕，动效编排完成）

### 质量门 G5：场景编排检查

| 检查项 | 通过标准 | 未通过处理 |
|---|---|---|
| 内容完整性 | s2-s6 每个场景都有 chip + 示意图 + card + callout + ghost | 补齐缺失元素 |
| 动效四原则 | 每场景 ≥3 种 ease、≥2 个入场方向，有错峰节奏 | 重新编排入场动画 |
| 停留帧规范 | 关键信息有停留时间，环境动效有且仅一种 | 加停留帧，统一环境动效 |
| 音画同步 | 入场时机与旁白对齐，讲到哪哪出现 | 调整 offset 时间 |
| GSAP 硬规则 | 用 fromTo 不用 from，退场禁用，环境动效挂 tl | 按 video-authoring.md 修正 |

> 动效设计原则详见 `references/motion-design.md`。
> GSAP 硬规则详见 `references/video-authoring.md` 第四节。

## 配音引擎说明（全平台兼容，零配置开箱即用）

### 跨平台兼容性一览

脚本采用**五级 fallback 链**，自动检测当前系统可用的最佳引擎，优先级从高到低：

| 优先级 | 引擎 | 支持系统 | 音质 | 是否需要配置 | 说明 |
|:---:|------|---------|------|:----------:|------|
| 1 | **Minimax TTS** | 全平台 | 高清 | 需要 API Key | `MINIMAX_API_KEY` 环境变量存在时自动启用，质量最好 |
| 2 | **macOS say** | macOS 仅 | 一般 | 零配置 | Tingting 中文女声，系统自带 |
| 3 | **Windows SAPI** | Windows 仅 | 一般 | 零配置 | PowerShell + System.Speech，系统自带 |
| 4 | **pyttsx3** | 全平台 | 一般 | 需 pip 安装 | `pip install pyttsx3`，跨平台离线 TTS |
| 5 | **纯文本时长估算** | 全平台 | 无音频 | 零配置 | 按字数估算时长（中文 4 字/秒），生成静音占位音频 + 完整时间轴，兜底模式 |

> **auto 模式（默认）**：自动按上表优先级选择第一个可用引擎。运行时会打印当前系统可用的完整 fallback 链，方便排错。

### 各引擎详解

#### Minimax TTS（全平台，推荐）

- **触发条件**：设置了 `MINIMAX_API_KEY` 环境变量时自动启用
- **音色选择**：女声 `female-chengshu`（成熟知性）/ `presenter_female`；男声 `male-qn-jingying` / `presenter_male`
- **配置方式**：`export MINIMAX_API_KEY=你的key`，然后正常运行脚本即可
- **适用场景**：正式发布、追求更高音质和表现力
- **可选环境变量**：
  - `MINIMAX_GROUP_ID` — 部分账号需要，作为查询参数
  - `MINIMAX_API_HOST` — 默认 `https://api.minimaxi.com`
  - `MINIMAX_TTS_MODEL` — 默认 `speech-02-hd`

#### macOS say（零配置，macOS 专属）

- **零配置**：不需要任何 API Key，macOS 系统自带
- **音色**：Tingting（中文女声，清晰自然）
- **适用场景**：快速预览、教学内容验证、无 API Key 时的完整产出

#### Windows SAPI（零配置，Windows 专属）

- **零配置**：Windows 系统自带 PowerShell + System.Speech 程序集，无需安装任何东西
- **实现方式**：通过 PowerShell 调用 `System.Speech.Synthesis.SpeechSynthesizer` 生成 WAV，再用 ffmpeg 转 MP3
- **音色**：系统默认中文语音（取决于 Windows 语言包）
- **适用场景**：Windows 用户零配置快速预览

#### pyttsx3（跨平台，可选安装）

- **安装**：`pip install pyttsx3`
- **特点**：纯 Python 离线 TTS 库，跨平台（macOS / Windows / Linux）
- **适用场景**：Linux 环境或不想用系统 TTS 时的离线方案

#### 纯文本时长估算（全平台兜底）

- **零配置**：永远可用，不需要任何引擎
- **实现方式**：按中文字数估算时长（约 4 字/秒），用 ffmpeg 生成静音 MP3 占位文件
- **产出**：完整的时间轴 `durations.json` + 静音音频文件，下游管线完全兼容
- **适用场景**：Linux 服务器环境、快速搭骨架、后续替换真实音频

## silent（无声）模式怎么工作

`index.html` 的 `<html>` 声明 `mode` 变量。渲染时 `--variables '{"mode":"silent"}'` 切到无声版：

- **时间轴**：每段等长紧凑（`SILENT_SEG`=4.6s，共 ~32s），不依赖音频长度
- **无字幕**：`MODE !== "silent"` 才生成字幕
- **不淡出**：最后一段不淡出到底，循环首尾无缝
- **质感动效全保留**：渐变/颗粒/波形流动都在（MP4 有帧间压缩，不像 GIF 怕连续色调和运动）

> 为什么不出 GIF？GIF 无帧间压缩，完整 7 段想要 720p 高清必然 >2MB（实测），只能降到
> 480p 且小字发糊。公众号早已支持内嵌 MP4 自动播放，无声 MP4 = 1080p 完整 + 8~10MB，
> 比任何 GIF 都清晰。所以动图产出用无声 MP4，不用 GIF。
