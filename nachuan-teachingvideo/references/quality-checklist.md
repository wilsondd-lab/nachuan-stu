# 质量检查清单

> 纳川教学视频的质量保证体系。**每个视频产出前必须通过以下质量门，不是"可以参考，而是必须检查。**
> 来源：CHAI 研究 (CMU/Harvard)、Mayer 多媒体学习原则、WCAG 可访问性标准、
> YouTube Creator Academy、专业制作质量标准

---

## 质量门与八阶段管线对应表

| 质量门 | 对应阶段 | 性质 | 阻塞性 |
|---|---|---|---|
| G0 研究深度检查 | 阶段 0：研究与拆解 | 预检查 | 软性阻塞 — 知识点错误不能往下走 |
| G1 概念提案检查 | 阶段 1：概念提案 | 方向对齐 | 软性阻塞 — 方向错了越走越偏 |
| **G2 脚本质量门 | 阶段 2：脚本撰写 | **第一道硬质量门** | **硬性阻塞 — 脚本烂，视频就烂** |
| **G3 五维分镜检查** | 阶段 3：分镜设计 | **第二道硬质量门** | **硬性阻塞 — 分镜不清，画面必乱** |
| G4 资产完整性检查 | 阶段 4：资产准备 | 技术检查 | 阻塞 — 缺文件渲染必崩 |
| G5 场景编排检查 | 阶段 5：场景编排 | 质量检查 | 阻塞 — 动效乱，观感差 |
| G6 渲染质量检查 | 阶段 6：渲染合成 | 技术验证 | 阻塞 — 技术不过不能交付 |
| 最终 QA | 阶段 7：质量验收 | **最终交付门** | **硬性阻塞 — 严重问题零容忍 |

> **规则**：未通过质量门时，返回上一阶段修正，不要硬着头皮往下走。
> 每道门最多 2 轮修正，第 3 轮仍不过则记录问题后继续（防止无限循环）。

---

## 一、脚本质量门（G2 — Storyboard Quality Gate）

**阶段**：阶段 2 脚本撰写完成后，进入阶段 3 分镜设计前
**性质**：第一道硬质量门 — 脚本是视频的骨架，脚本烂了后面再怎么救都没用

### 1.1 知识点准确性

- [ ] 核心概念的定义准确，与教材一致
- [ ] 所有数字、公式、单位正确（声速 340m/s、光速 3×10⁸m/s 等）
- [ ] 没有科学性错误（特别注意：物理定律、化学方程式、生物过程）
- [ ] 术语使用规范，符合对应学段的教材表述
- [ ] 如果用了类比，类比是准确的（没有误导性的过度简化）

### 1.2 结构完整性

- [ ] 7 段结构完整（1 引入 + 5 子概念 + 1 总结）
- [ ] 子概念按逻辑顺序排列（是什么 → 怎么工作 → 有什么特性 → 怎么用）
- [ ] 每段只讲一个点，没有塞入过多内容
- [ ] 首段有钩子（问题/生活场景/反直觉事实）
- [ ] 末段有收束（总结 + 回扣钩子）
- [ ] 段落之间有"但是/所以"的逻辑连接，不是"然后然后然后"

### 1.3 旁白质量

- [ ] 每段 30-50 字（过长会拖慢节奏，过短讲不清）
- [ ] 口语化，像老师在讲，不是念课本
- [ ] 用了"同学们看""你发现没有""这就是为什么"等口语连接词
- [ ] 不读公式：画面有公式，旁白用口语解释（"v=340m/s" → "每秒能跑 340 米"）
- [ ] 数字的读法写进文案（"3×10⁸" → "三亿米每秒"）
- [ ] 没有长难句，都是短句
- [ ] 语速估算：中文字数 ÷ 4.5 ≈ 秒数，与 min_duration 匹配

### 1.4 钩子有效性

- [ ] 前 5 秒能抓住注意力吗？
- [ ] 钩子类型与主题匹配（问题型、反直觉型、数字冲击型……）
- [ ] 钩子的问题在视频结尾有回答（首尾呼应）
- [ ] 没有用"今天我们学习……"这种无效开头

### 1.5 概念密度检查

- [ ] 全片新概念数量 ≤ 5 个（60-90 秒视频）
- [ ] 每个新概念有至少 10 秒的讲解时间
- [ ] 没有塞入超出学段的内容（初二的视频不出现高中概念）
- [ ] 新概念出现时都有明确的信号引导（"关键在于……""注意这里……"）

## 二、分镜质量门（G3 — Scene Plan Quality Gate）

**阶段**：阶段 3 分镜设计完成后，进入阶段 4 资产准备前
**性质**：第二道硬质量门 — 五维分镜不清，场景必然混乱

> **五维分镜的来源**：纳川教学视频原创的五维分镜框架
> 

### 2.1 五维分镜检查清单

每个场景（s1-s7）逐一检查以下五个维度。
**这不是"有时间再看"的参考——是每个场景都必须过的硬检查。**

| 维度 | 来源 | 检查项 | 通过标准 |
|---|---|---|---|
| **Subject 主题** | 纳川五维 | 画面的核心主体明确吗？与旁白一致吗？ | 视觉主角就是旁白正在讲的东西 |
| **Motion 动效** | 纳川五维 | 主体怎么动？动效服务于理解吗？ | 每个动效都有教学目的，不是装饰 |
| **Scene 场景** | 纳川五维 | 布局/叠加层/环境清晰吗？ | 一眼能看出场景结构，信息块 ≤3 个 |
| **Spatial 空间构图** | 纳川五维 | 元素位置关系合理吗？焦点明确吗？ | 视觉焦点与知识重点一致 |
| **Camera 镜头** | 纳川五维 | 视角和取景有明确意图吗？ | 固定/变焦/平移都有理由，不是乱动 |

### 2.2 场景内容完整性

每个子概念场景（s2-s6）必须有：

- [ ] section-chip（左上角编号 + 小节名）
- [ ] SVG 示意图（≥3 个图形元素）
- [ ] info-card（2-4 行要点 + 可选公式）
- [ ] 至少 1 条 callout 标注（指哪讲哪）
- [ ] ghost 底纹编号（与 chip 同号）

### 2.3 Animatic 思维规划法

> 在写场景之前，先用"动态分镜"的思维过一遍：如果这个场景是你在黑板上讲，
> 你会先写什么、再画什么、最后圈出什么？动画顺序要跟这个自然讲解顺序一致。

检查方法：

- [ ] 入场顺序符合"讲解逻辑"吗？（先框架后细节，先问题后答案）
- [ ] 是逐步揭示的，还是一次性全出来的？（应该逐步揭示）
- [ ] 讲到关键信息时，对应的视觉元素有强调吗？（高亮、放大、脉动）
- [ ] 如果关掉声音，只看画面，能大致看懂在讲什么吗？

### 2.4 视觉一致性

- [ ] 所有场景用同一套配色（主色、中性色、背景色一致）
- [ ] 字体风格统一（标题宋体、正文黑体，不混用）
- [ ] 卡片样式一致（kicker 小标、左侧主色线、圆角大小）
- [ ] SVG 线宽统一（4-8px，不出现 1px 的细线）
- [ ] 元素尺寸符合视频规范（标题 ≥130px、正文 ≥30px）

### 2.5 有限转场家族原则

- [ ] 全片转场类型 ≤ 3 种
- [ ] 模糊叠化是默认转场（占 80% 以上）
- [ ] push-up 转场 ≤ 2 次，且只用在有"章节推进"语义的地方
- [ ] 转场时长统一（模糊叠化 0.5s、push-up 0.5s）
- [ ] 没有使用花哨转场（3D 翻转、旋转、爆炸等）

## 三、最终 QA 清单（阶段 7 — Final QA Checklist）

**阶段**：阶段 6 渲染合成完成后，阶段 7 质量验收
**性质**：最终交付门 — 严重问题零容忍，必须全部通过才能交付

### 3.1 渲染质量

- [ ] 视频能正常播放，没有花屏、卡顿
- [ ] 分辨率 1920×1080，帧率正常
- [ ] 文件大小合理（60-90s 视频约 20-50MB）
- [ ] 没有渲染伪影（文字边缘锯齿、色块断层）
- [ ] 蒙太奇图（montage.png）中 7 个场景都有内容，没有空场景

### 3.2 字幕同步

- [ ] 字幕与旁白完全对应（没有错字、漏字）
- [ ] 字幕出现时间与旁白开始时间一致
- [ ] 字幕消失时间与旁白结束时间一致（稍晚 0.2-0.3s）
- [ ] 字幕位置在底部安全区内，不被遮挡
- [ ] 字幕字号 ≥34px，清晰可读

### 3.3 配色合规

- [ ] 正文与背景对比度 ≥ 4.5:1（WCAG AA 标准）
- [ ] 大标题与背景对比度 ≥ 3:1
- [ ] 主色只在强调元素上使用（不超过画面的 10%）
- [ ] 中性色与主色的色温相反（暖主色配冷中性，冷主色配暖中性）
- [ ] 没有颜色太接近导致区分不清的情况

### 3.4 文字与布局

- [ ] 所有文字在安全区内（不贴边、不出画）
- [ ] 没有文字重叠（inspect 的 overlap warning 全部处理）
- [ ] 点的字母标注和中文说明错开方位（不叠在一起）
- [ ] 字幕不遮挡重要画面内容
- [ ] 章节角标在右上角，不与其他元素冲突

### 3.5 动效流畅度

- [ ] 入场动画流畅，没有卡顿或跳变
- [ ] 转场自然，没有突兀感
- [ ] 环境动效循环正常（波形流动、呼吸脉动）
- [ ] 没有元素闪烁或忽隐忽现（seek 相关 bug）
- [ ] 最后一段的淡出自然（视频模式），或循环无缝（无声模式）

### 3.6 音频质量

- [ ] 旁白清晰可懂，没有杂音
- [ ] 音量适中（-14 LUFS 左右，峰值不超过 -1.5 dBTP）
- [ ] 没有爆音或削波
- [ ] 各段音量一致（没有一段大声一段小声）
- [ ] 音频与画面同步（嘴型不需要，因为是配音，但字幕要同步）

## 四、音频设计标准

### 4.1 旁白质量标准

| 指标 | 目标值 | 说明 |
|---|---|---|
| 语速 | 130-150 字/分钟 | 教学场景偏慢，给思考时间 |
| 音量 | -14 LUFS 集成响度 | 平台标准（YouTube/抖音） |
| 峰值 | ≤ -1.5 dBTP | 避免爆音 |
| 清晰度 | 高 | 每个字都能听清 |
| 情感 | 亲切、鼓励 | 像耐心的学长/学姐 |

### 4.2 TTS 引擎选择指南

| 引擎 | 音质 | 配置 | 适用场景 |
|---|---|---|---|
| **macOS say (Tingting)** | 中等 | 零配置 | 快速预览、无密钥环境、教学内容验证 |
| **Minimax (female-chengshu)** | 高 | 需要 API Key | 正式发布、追求高质量 |

**切换方式：**
- 默认（无配置）：自动用 macOS say
- 设置 `MINIMAX_API_KEY` 环境变量后：自动升级为 Minimax
- 也可以用 `--provider say` 或 `--provider minimax` 强制指定

### 4.3 语速与时长的关系

中文字数与音频时长的大致对应：

| 字数 | 预估时长（say） | 预估时长（minimax） |
|---|---|---|
| 30 字 | ~6.5s | ~6s |
| 40 字 | ~8.5s | ~8s |
| 50 字 | ~10.5s | ~10s |
| 60 字 | ~12.5s | ~12s |

公式：时长(秒) ≈ 字数 ÷ 4.5（say）≈ 字数 ÷ 5（minimax）

**检查方法：** 生成音频后核对 durations.json 中的 audio_duration，
如果某段明显偏长（超过 12s），考虑精简文案。

### 4.4 背景音乐建议（可选）

如果后续添加背景音乐：

- 类型：Lo-fi / 轻钢琴曲 / 氛围音乐
- BPM：90-110（平静但不沉闷）
- 音量：比旁白低 18-20 dB（旁白为主，音乐只是打底）
-  EQ：切掉 2-4kHz 频段，给人声让路
- 注意：教学视频默认不加 BGM，避免分散注意力；如需加，仅限开头结尾

## 五、自检流程（八阶段对应）

每次产出一个视频，按以下顺序自检。每一步不通过就回去修正，不要往下硬冲。

### 第 1 步：研究自检（阶段 0 后）
1. 核心定义查过教材了吗？ — 准确性是底线
2. 找到至少一个常见误解了吗？ — 误解优先法的弹药
3. 5 个子概念拆解逻辑通顺吗？ — 从是什么到怎么用

### 第 2 步：提案自检（阶段 1 后）
1. 钩子 5 秒内能抓住注意力吗？ — 制造信息缺口了吗？
2. 弧线类型跟内容匹配吗？ — 是误解优先还是引导发现？
3. 开头的问题结尾能回答吗？ — 首尾呼应了吗？

### 第 3 步：脚本自检（阶段 2 后，G2 质量门）
1. 读一遍旁白，计时 — 每段 30-50 字吗？
2. 数一数有几个新概念 — 超过 5 个了吗？
3. 首尾呼应吗？ — 开头的问题结尾回答了吗？
4. 科学准确吗？ — 翻教材核对关键知识点
5. 是"但是所以"不是"然后然后"吗？ — 逻辑连接对了吗？

### 第 4 步：分镜自检（阶段 3 后，G3 质量门）
1. 每个场景的五维都明确了吗？ — Subject/Motion/Scene/Spatial/Camera
2. 关掉声音能看懂大概吗？ — 画面自己能讲故事吗？
3. 全片转场不超过 3 种吗？ — 模糊叠化是默认吗？
4. 入场顺序符合讲解逻辑吗？ — animatic 思维过了吗？

### 第 5 步：编排自检（阶段 5 后，G5 质量门）
1. `npx hyperframes lint` — 0 error 吗？
2. `npx hyperframes inspect` — overlap warning 都处理了吗？
3. 每场景 ≥3 种 ease、≥2 个入场方向吗？ — 不是全 y+opacity 吧？
4. 蒙太奇预览 — 7 个场景都饱满吗？配色统一吗？

### 第 6 步：成片自检（阶段 7，最终 QA）
1. 完整看一遍视频 — 有哪里卡、哪里糊吗？
2. 关掉声音看 — 只看画面能看懂大概吗？
3. 听声音不看画面 — 旁白清晰流畅吗？
4. 看蒙太奇图 — 7 格都正常吗？

### 通过标准

- G2 脚本质量门：全部 ✅
- G3 分镜质量门：≥4/5 维度通过，关键维度（Subject 主题一致性、Motion 动效有效性）必须通过
- 最终 QA：严重问题为 0，轻微问题 ≤ 2 个
